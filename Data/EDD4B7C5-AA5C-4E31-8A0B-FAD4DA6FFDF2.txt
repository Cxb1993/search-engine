背景說明： 
 
近年來，手機已經成為十分普及的產品，也越來越多手機整合了高階的處理器，
使得手機具有越來越高的運算能力，加上網際網路的普及，手機的網路傳輸能力也漸
次增強。因此，手機的網路應用，較之以往更為可行。 
在各種手機網路應用中，行動遊戲一直是眾所矚目的焦點。受惠於寬頻網路及行
動通訊應用服務的逐漸普及，行動遊戲市場的未來發展相當看好。 
根據上述市場趨勢研究，我們在本計畫中，將著眼於手機行動遊戲的應用。在研
究手機行動遊戲開發之前，需要先瞭解手機在遊戲開發上的一些重要限制，這些限制
可從硬體及軟體兩方面來檢視。首先是硬體部分，雖然手機的硬體能力已經較以往提
昇許多，但它的運算和顯示能力，跟個人電腦或遊戲機相比，仍然相去甚遠。底下列
出影響手機遊戲開發的主要硬體限制： 
 螢幕尺寸和解析度：由於方便攜帶的需求，手機通常設計成能夠放入口袋，這當
然影響了螢幕的尺寸與解析度。早期大部份的手機螢幕解析度都在 90×120 像素
至120×160像素之間。目前的主流照相手機的螢幕解析度則已經提升至240×320。
而一些高階手機支援更高的解析度。不同手機支援各式各樣不同的解析度，更提
升設計通用手機遊戲介面的難度。 
 使用者輸入：手機的輸入裝置主要用來撥打電話，因此大多數僅支援 ITU 鍵盤（數
字按鍵、上下左右瀏覽鍵及其他少數的撥號、選單按鍵），少數支援觸控螢幕及
手寫輸入。因此，如何利用手機有限的輸入方式，設計出讓遊戲玩家覺得方便又
容易操作的介面，需要遊戲設計者仔細考量。 
 記憶體限制：目前市面上手機的記憶體容量差異相當大，從數 MB 到數 GB 都有，
一般經濟型手機的記憶體容量通常僅有數 MB。為了開發適合多數手機進行的遊
戲，也需要考量記憶體的限制。 
 計算能力：目前大部份手機上的處理器(CPU)的速度，也有相當大的差異，大致
上介於 100 到 500MHz 之間。因為處理器速度的影響，計算能力在不同手機上可
能會產生相當不同的結果。如果應用程式需要大量的計算時，在較低階的手機上
執行的速度就會變慢。 
上述硬體限制，記憶體與處理器運算能力皆能與時俱進。最主要的限制，是在螢
幕大小與輸入方式的演進較慢。對於軟體開發而言，在設計使用者操作介面上要花費
較多時間。 
接下來，討論軟體開發方面的限制。最初，不同廠商推出的手機，通常搭配不同
的作業系統。近年來，由於手機的普及，種類越來越多，已有廠商聯合推出能夠通用
的作業系統。目前的智慧型手機作業系統主要有 Symbian、Windows CE、PalmOS 及
Linux。其中又以 Symbian 與 Windows CE 兩種作業系統分佔市佔率較高。我們將此開
發環境分為「作業系統相依的開發平台」與「跨作業系統的開發平台」兩類。對於遊
戲開發業者而言，為了支援不同的作業系統及不同的開發平台，需要投入更多人力成
本解決移植、相容性等問題，因此也提高整個開發成本。這些問題，必相對減少對於
遊戲內容的開發資源。但是豐富的遊戲內容才能吸引使用者，因此，如何減少支援多
平台時所需的人力以縮短遊戲發展時程，成為一個重要的課題，也是本計畫的主要目
 GUI 模組：為了支援遊戲畫面的開發，我們將提供基本的視窗元件，包含 Frame 
Window、Label、Button、Menu、Image 等。除了視窗元件外，我們也設計繪圖用
的 API，封裝於 Graphics類別中。由於採用事件驅動模式，所以我們也定義下列
事件：Key事件、Point事件、Button/Menu 事件及 Draw事件。 
 Network 模組：Network 模組封裝底層 Network 複雜的設計，提供簡便的函式，
利用事件驅動方式通知 Application 已收到網路資料。我們預計實作一雛形模組
支援 TCP/IP 以及 Bluetooth的通訊協定。 
 
遊戲流程： 
 
我們設計一框架支援標準的遊戲流程，並提供高階的 API 讓遊戲開發者使用。依
據群想網路科技所提供的資料，將遊戲流程分為四個階段: 
 階段 0 初始化：當程式開始執行時，第一件事就是要辨明現在使用的作業系統，
產生對應的物件。在此階段產生的元件主要會有 Application 和 GUI。 
 階段 1 設定遊戲：進入遊戲程式本體，螢幕展示歡迎畫面，程式內部也要去檢索
系統資源情況，還要提供使用者介面給使用者操作，例如登入、選擇連線方式等。
在使用者選擇連線方式後，就進入連線登入畫面。等連線完成，即開始進行遊戲。 
 階段 2 進行遊戲：可細分為主動與被動兩種階段。主動階段表示輪到使用者的回
合，此時使用者可以輸入選擇，像是下棋、選擇撲克牌等。被動階段則代表輪到
網路端對手的回合，此時大部分的使用者介面會被封鎖，等待對方完成輸入。 
 階段 3 結束遊戲：在不斷往返的階段 2後，遊戲有了勝負結果，來到了結束階段。
此時會先顯示結束畫面，再來要提供結束時的選項，看是要再來一盤（回到階段
2）、到歡迎畫面（回到階段 1）、還是結束程式。 
 
遊戲相關 API: 
 
Application 模組中增加 State和 Rule類別:  
 State 類別：State 類別是把遊戲中的資料、玩家的設定和系統資訊集中起來，
可以分成幾類：私人數據﹝用戶名、密碼、積分﹞、系統數據﹝使用平台、螢幕
大小、記憶體容量、網路類型﹞、遊戲數據﹝回合、得分、棋盤情況﹞。 
 Rule類別：這個類別定義出和遊戲規則有關的函式，像是 CheckMove﹝確認此步
合不合法﹞、IsGameOver﹝確認遊戲結束沒﹞、GetWinner﹝回傳勝利者﹞。從
另一角度看，遊戲主要邏輯就是在這些函式中完成。
