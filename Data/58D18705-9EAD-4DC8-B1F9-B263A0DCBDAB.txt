摘要 
隨著網際網路快速發展，網路上所累積的資訊也越來越多，一般來說，使用者常用的
方法就是透過搜尋引擎來尋找自己想要的資訊，但是資訊累積量多、維護不易，使得搜尋
結果中充斥著許多並非使用者真正想要或鏈結失效的資訊，本研究首先以 RSS 為基礎，希
望能建構一個高效能之資料蒐集與資訊整合系統。主動收集目前存在於網際網路上的 RSS 
並加以整理分類，讓使用者方便搜尋以及查閱，接著提供內容整合功能，在各個不同的應
用中，整合一份對使用者來說是真正需要且有用的資訊，亦可藉此達成資訊客製化的目標。
如何有效管理資訊並達到資訊的重用性，是當今數位內容領域中一個重要的開發議題。針
對此問題，本計劃另提出了資訊物件模型的概念。ㄧ個資訊物件包含了因應在應用上所需
之基本的資料單元，包括資料部份及處理這些資料的方法，其中也包含描述此資料的格式。
處理資訊物件方法之建構應具有擴充性以因應不同使用者及未來不同的需求在資訊物件
中，將資料及其處理方法建構成一個單元，有利於資訊的儲存、查閱、交換及重用等需求，
也有利於長期保存資訊之解讀及處理問題，而為了將資訊物件作有效率的管理與儲存，計
劃中亦規劃了一XML Document Repository，提供長期資料的管理與維護。 
    最後本研究提出一套系統模型，其目的在於利用前述的資訊物件與XML Document 
Repository的設計，有效提供行動使用者感興趣之相關資訊。此模型包含三個重要的物件，
包括位置資訊物件(Location Object)、 地圖物件(Map Object)及Point of Interest(POI) 
Object三部分。位置資訊物件代表一個區域性範圍資料，此資料為資訊消費者所感興趣的
資訊。位置資訊物件包含一個任意的Map Object和許多的POI Objects。地圖物件所代表可
以是一張地圖或是一張圖片，此地圖物件根據區域性範圍的關係和許多POI Object關聯在
一起。每個POI Object儲存使用者感興趣標示點資料，此資料包括標示點所歸屬領域的資
訊。本系統所提供的模型在於便利使用者可簡便地取到所感興趣之位置相關資訊，假如使
用者有配置位置感知裝置，則系統更可主動與即時提供使用者位置資訊。 
關鍵字：RSS、資訊客製化、發佈/訂閱系統、資訊物件、XML Document Repository、位置
資訊物件、地圖物件、POI、位置感知裝置

2. XEBS

系統中並使用 XML Object 的概念，提供自給自足、完整的資訊個體，由於 XML 具有明

5. RSS(Really Simple Syndication) 
 
什麼是 RSS 
RSS 是由 Netscape 在 1999 年所開發，它代表三種可能的縮寫: RDF Site Summary、
Rich Site Summary 或 Really Simple Syndication；RSS 是一種以 XML 為基礎的描述
格式，主要用來整合(syndicate)各網站超連結的清單，並藉由附屬的一些 metadata
資訊讓資訊消費者能決定是否要進一步瀏覽完整的詳細內容。對於網站開發者(Web 
developer)而言，特別像是以發佈新聞為主的網站(news-like site)或是個人網頁空
間(Weblog)，能利用它來描述與整合網站內容以建立出屬於自己的 RSS feed(也就是一
個符合 RSS 規格的檔案)。當然他不只是用在新聞發佈上，任何 list-oriented 
information，只要能將資訊內容以列項(itemize)的方式表達就能透過 RSS 將他們整
合起來後提供給資訊消費者；像是 wiki 中哪些網頁是”最近改變”的、CVS 的
changelog 或甚至是一本書的改版或校訂的歷史，而 RSS 也不一定只能應用在 Web page
上，像是 Instant messaging 服務就適合利用它來傳送 headline，如同手機上的簡訊
服務。

(二)、系統規劃 
本 RSS 資料搜集、查詢與整合系統規劃兩大資料庫與五大模組，下圖 7為系統架構圖。 
 
圖 7、系統架構圖 
系統管理者可透過資料庫管理模組對兩大資料庫做新增、刪除與修改，資訊提供者與
網路使用者亦可透過資料庫管理模組提供網際網路上 RSS 文件的網路位置。RSS URL 
Database 內存放的即是 RSS 文件相關資訊，如分類、提供者(來源)、網路位置，這些資訊
將提供給資訊擷取模組，進行資訊擷取與剖析，剖析後的 RSS 文件就存放在 XML Document 
Repository 之中，資訊擷取模組又分為定時擷取與即時擷取，主要是因應比對模組與整合
模組不同的需求而設計。比對模組主要功能是讓使用者對系統內收集的 RSS 文件做關鍵字
比對搜尋，比對模組啟動後會觸發資訊擷取模組的即時擷取，先擷取目前網路上最新的 RSS
文件，再搜尋 XML Document Repository 內的 RSS 文件。整合模組則分為兩種資訊整合方
式，一種是多重比對，類似比對模組功能，但相對於比對模組只能對單一一種應用做搜尋，
標三部分(如圖 8)，如此設計在於提供系統可分別滿足兩種類型的使用者作相對應的運作
處理。 
 使用者資料(User Info) 
主要記錄使用者的名稱及使用者的類型。 
 使用者意圖(Intention) 
主要提供資訊消費者可以自己選擇主題設定，滿足不同的應用需求。 
Stationary user 的喜好主題設定，如旅遊導覽、醫療電子書 
Mobile User 的喜好主題設定，如動態旅遊、醫生巡房記錄 
 追蹤使用者動態坐標(Dynamic Location)  
主要是針對具有行動通訊裝置的 Mobile User 所設計，追蹤使用者的即時坐標資
料，提供給系統做使用者動態位置的監控，此目的在於即時的通報展現資料給使用
者。 
 
地圖物件 
系統提供影像圖片與標示點為基礎做為特定主題的展現，對於影像圖片的展現方式及
規格，設計了地圖物件模型(Map Object) ，主要的應用功能在於幫助系統儲存所要展現影
像圖片的地理坐標位置、圖片規格及標示點定位的資料，此物件模型包含 System 
Metadata、Text Data 及 Map Method 三部分。 
 
 
圖 9、地圖物件模型的圖示 
 System Metadata : 後端資料庫為了維護每個 Map Object 的內容資料所給予的描
述資訊。 
 Map 主要是地圖或影像，Text Data 是此地圖或影像文字的描述資料。 
 Map Method :提供標示點定位所需的座標轉換公式。決定每個標示點(POI)在影像
圖片中相對座標的運算公式。 
 
POI 物件 
POI 物件模型的概念，是如何來包裝 Publisher 所發佈的 Event，此 Event 為 Publisher
感興趣的標示點資料，主要的應用功能在於儲存 Publisher 所發佈的每個標示點所代表標
示點的詳細資料、標示點實際或相對的定位資料及圖示說明資料。所以設計 POI 物件模型，
將 POI 看成是一份完整的資訊單元，其中不僅包含了位置資訊，也記錄每個 POI 在其所歸
屬的區域性範圍內所代表屬性。此 POI 物件模型包含 System Metadata、Text Object 及
POI Method 三個部份。 
 Subscribe Rule(訂閱條件) 
主要是記錄使用者向系統訂閱所需區域性相關資料的條件，包含兩種的訂閱條件: 
1. Primary Subscription Rule : 找出所有滿足條件的 Objects。 
2. POI Filter Rule :對 POI Object 做進一步特定種類做篩選。 
 Map Object  
代表應用中所包含要展現區域性範圍的影像圖片或地圖、地理坐標位置、圖片規格
及標示點定位的資料。 
 POI Object Set 
包含許多 POI Object 的集合，而 POI Object 應用在於包裝儲存每個標示點所代表
標示點的詳細資料、標示點實際或相對的定位資料及圖示說明資料。 
 
對於資訊消費者類型的不同，所訂閱的條件也不相同，對於 Map Object 及 POI Object
的需求相對的也不相同，依不同的使用者類型如 Stationary User 及 Mobile User 都可套
用行動資訊物件模型。 
 
Location Sensor Module 
系統的思考規劃中，包含有行動資訊消費者(Mobile User)的規劃，系統必須取得行動
資訊消費者的即時位置資訊，才能進行比對其地域性相關的資料。在系統的規劃中，建立
Location Sensor Module 來處理行動資訊消費者的動態位置改變，即時比對動態位置資訊
是否與使用者所預先訂閱的位置資訊相同，再回報給系統其他的 Module 做進一步的系統處
理。 
市售的位置感知裝置(如 GPS、RFID、WLAN 等)都不相同，使用不同的感知裝置其傳輸
的協定及資料格式都不相同，所以在 Location Sensor Module 的設計上，特別的思考此問
題，在 Location Sensor Module 中設計 Sensor Signal Detector，此設計的目的在於判
斷感知裝置的傳輸格式，再交由各別不同裝置的 Device Monitor Tracker 做監督的工作，
如圖 12，如此系統在搭配不同的位置感知裝置時，更具備有彈性。 
 
圖 12、Location Sensor Module 設計概念圖示 
系統搭配行動感知裝置，提供給使用者不受地區的限制，及更廣泛的應用，加上
Location Sensor Module 的管理，達到擴展使用更多不同裝置的好處。
