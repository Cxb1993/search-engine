摘要 
本計畫為自由軟體總計畫『基於開放嵌入式系統之智慧型車用「視聽娛樂」與「行車
安全」網路服務中介平台之設計與實作』項下之子計畫二『為車用多媒體通訊而設計之調
適性行動視訊串流技術之設計與實作』，主要之目的為針對行車環境中行動網路訊號的不
穩定性(如 Multi-path fading、Path Loss、Shadowing 等效應)與嵌入式設備硬體資源的限制
與特性，設計並開發一套適用於嵌入式作業系統下的車用視訊串流傳輸服務平台，以利上
層行動視訊串流相關應用服務之開發。整個計畫擬於三年 (98-100 年) 完成，我們將依據
行車中網路的特性，設計與實踐「 frame trimming / packet-based rate adaptation 
(micro-adaptation)」與「GAP-based ARQ / frame-based rate adaptation (macro-adaptation)」兩
項創新的調適機制，以促進視訊串流對行車環境變異的適應能力，希望藉由此視訊品質調
適機制，使系統各部分能迅速達到穩定且最佳的效能平衡，以避免bottleneck 的產生，使整
個調適過程能即時且順暢。此平台將遵照light-weight CMMI 的規範開發，以WinCE及開放
式Android 為系統開發平台；我們同時將透過在不同環境下的實際運作，收集相關實驗數
據，深入分析驗證這些機制的優劣性，並探討各項系統參數的設定。本子計畫所開發之系
統，將提供一套基礎的程式開發介面（API），供上層應用使用；本系統之原始碼亦可供產
學界在未來設計類似系統之參考；同時本平台將與其他子計畫緊密整合，提供完整之智慧
型車用多媒體通訊服務。 
關鍵詞： 調適性行動視訊串流、訊框修剪、速率調適 
Abstract 
This project, Design and Implementation of Adaptive Mobile Video Streaming (AMVS) 
Mechanisms for Vehicular Multimedia Communications, is Sub-project 2 of an integrated project, 
Intelligent Vehicular Entertainment and Safety Web Services Platform Based on Open Embedded 
System. The purpose of the project is to develop a mobile video streaming transport service for a 
vehicular environment, where the wireless mobile networks usually show unstable signal quality 
(due to the effects of multi-path fading, path loss, and shadowing) and vehicular embedded 
devices have limited resources. This platform is able to facilitate the development of upper-later 
mobile video applications. The project timeline covers three years, 2009-2011. We will design 
and implement two novel adaptive mechanisms, frame trimming / packet-based rate adaptation 
(micro-adaptation) and GAP-based ARQ / frame-based rate adaptation (macro-adaptation). 
Basically, they adapt video streams according to the network and system conditions in a rapid and 
smooth manner by keeping a balance among several system components and avoid the 
bottlenecks. We will follow the light CMMI standards to implement the system. The system will 
be implemented on a WinCE and an open-platform (like Android). We will conduct the system 
evaluation under various environments and provide guidelines for the parameter tuning. We will 
deliver a set of application programming interfaces (API) for upper-layer applications. The 
source codes will also be open to the public as a reference of future development of a similar kind. 
The platform will also be tightly integrated with the other sub-projects to provide a 
comprehensive intelligent vehicular multimedia communication services. 
Keyword: adaptive mobile video streaming, frame trimming, rate adaptation 
三、 研究方法 
本研究主要探討以視訊內容為主的調適機制。考慮的因素主要著重於 Home (Roadside) 
-to-Vehicle 的行動網路環境與嵌入式平台系統資源問題，為達到一具調適性服務品質保證之
串流傳輸服務且要達到 graceful adaptation，我們將 streaming adaptation 的策略依調適機制
運作的範圍與反應的快慢分成 macro adaptation 與 micro adaptation 兩種。 
 Micro-adaptation: 依據目前的網路環境封包的遺失狀況，做即時的微調反應，此
做法是為了解決無線網路容易受到干擾造成封包遺失，我們將先判別遺失的情
況，若屬於嚴重干擾才啟動 micro adaptation。 
 Macro-adaptation：依據較長期的系統硬體資源與網路狀況觀察後做適度的串流速
率調整，當判別網路發生壅塞時，則能在尚未發生嚴重的封包遺失前啟動
macro-adaptation。 
系統架構如圖一所示，adaptive mobile video streaming (AMVS)系統的實作中，會實作兩
個部分，第一部份為 server 端，本計畫命名為 adaptive mobile video streaming gateway，以
下將簡稱為 AMVSG。AMVSG 除了針對視訊影像不同種要等級的辨識之外，同時提供 h.264
影音的串流 micro/macro 調適策略。第二部分為 client 端，本計畫命名為 adaptive mobile video 
streaming adaptor，以下將簡稱為 AMVSA。AMVSA 為行動手持裝置撥放 h.264 影音串流，
主要是將破碎的影格剔除並監控網路環境狀況。

除了提供 packet-based 的調適之外，第一年後期我們著手研究 frame-based 的調適。當
網路傳輸進入一個常態後（如：從 3G 切換成 GPRS），我們期望系統能快速解決壅塞並收
斂至一個新的平衡狀態。因此需針對原始影片作調適以符合有效的 throughput，但在不同的
網路環境下實際可用的有效頻寬不足以服務原始的影片基本傳送速率，因此串流影片需作
調適符合目前的網路環境以達穩定狀態。而調適的方法可分為下面三類： 
1. Temporal scalability，調整 frame rate。 
2. Spatial scalability，調整原來影片的 frame size 或 pixel 次取樣。 
3. SNR scalability，調整量化與 DCT 係數。 
第一年我們將採用 temporal scalability 的調適方式調整原來影片的 frame rate，透過捨
棄不重要的 frame 調整符合網路頻寬的 frame rate 換取更平順的播放效果，如圖三下半部；
此外我們也增加 GAP-based ARQ 重傳機制，讓在播放時效性內遺失的封包可以要求重新傳
送，降低封包遺失率。而 ARQ 的方法也分為三類：stop and wait、go-back N、select repeat。
我們將採用 select repeat 的作法，同時參考 SCTP-SACK 的方式設計來減少發出要求重傳的
封包。 
第二年（99 年）：Macro adaptation (frame-based rate adaptation)：主要設計結合時間域與
空間域 frame rate 調整的傳送服務。 
第二年我們將著重在 macro-adaptation 的議題。核心模組為 frame-based rate adaptation
機制。不同於第一年 temporal scalability 的 frame-based rate adaptation 機制，第二年我們更
深入的探討 H264/SVC 調適的特色，結合 spatial scalability 與 temporal scalability 的調適方
式，對原來影片的 frame size 或 frame rate 作動態調整，以符合不同網路環境甚至不同的終
端裝置。其中要如何隨著不同網路情況平順地切換到符合網路頻寬影片的相關設計都是值
得研究的議題。此外如何在 throughput cost 和視覺感知的質量之間取得一個平衡也是值得
進一步調查，這將會是我們第二年度的研究重點。

七、 計畫成果自評 
我們建議在無線通訊網路環境下，依據網路情況與影片內容研究動態調整視訊串流傳
輸。本計劃提供了兩個重要的優點：一、有效抑制視訊錯誤蔓延，二、降低封包遺失率。
我們獲得一些有關不同網路環境下流量控制的理論結果，提供動態的調整視訊串流傳輸，
並有效抑制視訊的錯誤蔓延與降低封包遺失率。 
此外我們觀察到當一直處於極低頻寬的網路環境下(如 GPRS)，temporal scalability的
frame-based rate adaptation 機制，會將影片刪減到只剩 I frame，甚至因為要能符合目前網
路的頻寬，有些I frame也得跟著刪減才行。雖然這樣能符合網路頻寬傳送，但也會因為刪
減過多frame而造成影片撥放不順的問題。 
所以要在異質網路環境下平順切換符合頻寬的影片也是一個有趣的議題，值得我們進
一步研究，例如採用H264/SVC空間域的調適概念調整不同解析度的視訊資料。如何在
throughput cost和視覺感知的質量之間取得一個平衡是值得進一步調查，這將會是我們下一
年度的研究重點。

 An Adaptive Suffix-Window Caching Scheme  
for CM Proxy Server 
 
YUAN-TSE YU 
Dept. of Software Engineering  
National Kaohsiung Normal University  
Kaohsiung City, Taiwan (R.O.C) 
yuyt@nknucc.nknu.edu.tw  
SHEAU-RU TONG 
Dept. of Management Information System 
National Pingtung University of Science and Technology 
Pingtung County, Taiwan (R.O.C.) 
srtong@mail.npust.edu.tw 
 
 
Abstract—the proxy server is generally deployed for accessing 
streaming server in order to serve amount of domain’s clients on 
Internet simultaneously. For continuous media (CM) services, in 
order to decrease the initial delay when a user request a VoD 
service, the proxy stores two kinds of segments: prefix and suffix; 
former one means the beginning segments of a video, which are 
allocated in advanced for decreasing initial delay, and latter one 
means the following segments, which are stored temporally for 
sharing segments between amount of users. Therefore, the study 
proposes an adaptive suffix-window scheme, which is able to 
dynamically schedule the storage allocation for suffix window in 
response to gain the optimal performance. The experimental 
results show that the proposed scheme significantly increases the 
number of served clients and improves the performance of proxy 
storage utilization when the client owns limited buffer. 
Keywords-component: Proxy server; continous media; video 
streams; adaptive suffix-window scheme. 
I.  INTRODUCTION 
With rapid development of Internet, the transmissions of 
continuous media (CM), e.g., video and audio, over network 
have widely used. However, the limited bandwidth and traffic 
congestions result in poor presentation quality at client slide. A 
proxy server is usually adopted in a local area network (LAN) 
to solve the above problems. When the frequency of on 
demand users increase on the specific videos, the proxy 
caching mechanisms [11][17][18] become the main issue for 
effecting the performance. Due to an abundant video segments 
occupying large buffer space of proxy, the proxy then caches a 
video as a single object for servicing single user, the proxy 
capacity would be easily exhausted. Therefore, an alternative 
method is that the proxy only caches partial segments of video 
and shares the segments between the users. There are some key 
issues should be take into account when designing a proxy 
caching mechanism.  
 Long initial delay: the processes of channel setup and 
resource reservation between server-to-proxy and proxy-to-
client usually take long time. Thus, the latency for mobile 
users with limited buffer, e.g., PDA and mobile phone, is 
not acceptable. Beside, the response time in LAN always 
smaller than that in WAN. Therefore, an alternative strategy 
is to utilize the cache in streaming proxy to decrease 
response time in WAN. 
 Restricted network bandwidth: the WAN bandwidth is 
typically limit and variation. In order to support amounts of 
users’ on-demand request, previous researches normally use 
a conventional transmission technique, which is to gather 
requests arrived at closed time points. These requests are 
configured as a multicast group and the proxy multicasts 
the streams to the clients in same group synchronously. 
Although clients request times are different, the 
multicasting streams are transferred with synchronous rate 
[9]. Thus, the system must provide an appropriate caching 
mechanism for proxy and clients, and the system can 
correctly transmit streams to each client with different 
arrival time [1][2]. 
 Clients with various buffer sizes: when a group of clients 
requests a video, they can obtain prefixes from proxy for 
decreasing the initial delay time. Simultaneously, the proxy 
requests and receives suffix from an Internet video server. 
After acquires suffix, the proxy multicasts the suffix to 
clients. Multiple clients can share the same stream. 
 Restricted resource of proxy server: in previous cache 
mechanisms, there were many mechanisms [13] of users 
requiring the Internet video server through proxy server to 
exchange data [3][4][12][16]. However, these mechanisms 
are not applicable to video system in real network 
environment because the video and audio data volume is 
quite massive, regarding it as a single object as cache 
exhausts the resource of Proxy Server. Furthermore, 
different videos have different request frequencies, if the 
request frequency of video is not considered in caching 
policy, which results in the videos having high request 
frequency cannot use the resource of proxy since the proxy 
server cached videos of low request frequency in advanced, 
so that the video server must transmit high-frequency 
videos over and over, and network traffic increases. 
In order to solve the aforesaid problem, this study proposed 
and designed a caching policy applied to proxy server for 
transmitting CM (see Fig. 1), the proxy caches the video 
through a flexible proxy resource managing mechanism after 
receiving the video from the Internet Video Server. The 
principle is to ensure videos of high requested frequency with 
higher priorities to use the resource of proxy and to cache more

According to the above discussions, this study designs a 
proxy caching mechanism to assist transmitting the CM object 
between client, proxy, and server, and the proposed mechanism

III. PROBLEM STATEMENT  
A. Problem Statement  
Assume that a group of users in network request for a video, 
the requests go through the proxy server firstly. The proxy 
server stores prefix segments of this video temporarily, and 
then the proxy directly transmits to these grouped users. 
Meanwhile, the proxy server downloads the suffix following 
prefix from the video server, and plays suffix after users saw 
the prefix, so as to avoid interrupted play. On the system 
structure, it is assumed that the user would keep viewing the 
video until the end, and the client do not leave the system 
during the playback. The user also requires that the prefix of 
each video can be found in the proxy. The transmissions 
between proxy and user comprise through unicast and multicast, 
whereas the transmissions between server and proxy are to only 
adopt unicast.  
Supposed that the VoD System is aimed to provide no 
initial delay service, which means the user is normally served 
as soon as logged in. The proxy stores the prefix of all videos, 
and many users over LANs typically require for their favorite 
videos. For the users located at the system, they can receive 
streaming data through unicast and multicast at the same time. 
The buffer of each user j is defined as bj, and the proxy buffer 
can store videos of S unit length.  
In conventional proxy caching schemes, the continuous 
initial portions of video stored in the proxy are called prefix. 
Because the initial video segments are often viewed frequently, 
the proxy caches prefixes to satisfy the user's request with 
shortest initial delay. The prefix-based algorithms comprise 
MPach and segment-based mechanisms. The following section 
shows related mechanisms, and the study proposes the ASW 
and ASW+ mechanisms, that as the proxy buffer caches the 
prefix and suffix. 
B. Research Method 
 MPatch caching algorithm 
Wang et al. [16] proposed the multicast past patching prefix 
caching (MPatch) method, which combined patching with 
SBatch. They hypothesized proxy-client transmission path is a 
multicast path. Let the threshold of a suffix be [vi, vi+ Gi]. 
Suppose client requests arrive at ti (0 -vi+Gi ), the proxy then 
transmits the suffix of a playing video via unicast to clients (in 
Fig. 4). If client requests arrive at ti (0–vi), a client receives 
segment[0,ti] via unicast from the proxy. If a request arrives at 
ti (vi< ti ≤Gi), the client uses a separate channel to obtain the 
patch without the prefix; that is, segment[vi,ti] is directly 
transferred via unicast from the CM server (in Fig. 5). 
Adopting suffix threshold with flexible stream merge results in 
better bandwidth utilization. Therefore, the disadvantage for 
both mechanisms is that a client must have sufficient buffer to 
cache material ti length. If buffer space is insufficient, merging 
clients’ streams do not work.

When arrival rate increases, meaning persons logged in 
unit time is relatively increased. Master user logs in and

processing, each Suffix Window serves a group of users.

serve more users via suffix window. When users with

E. Impact of Distribution (Di ) on network transmission 
volume 
This experiment probed into the selection probability of 
videos, and both of Uniform and Zipf distributions impact on 
the profit value of network bandwidth. With changing the 
selection probability of videos, the results show ASW and 
ASW+ still have preferable bandwidth profit value.  
Uniform distribution sets the selection probability of 
videos as equivalent, 20 videos are simulated in the system, 
the selection probability of each video is fixed at 1/20. The 
probability of using Zipf distribution for distributing videos is 
shown in Fig. 17, simulating the popularity of all videos 
distributed randomly every time. The test adjusted Zipf 
function parameters (K1 = 0.5, K2 = 0.1), so the video 
distribution would not have extreme situations (only some 
certain videos would be selected). Regardless of uniform or 
Zipf distributions, ASW and ASW+ mechanisms can 
obviously reserve WAN bandwidth. Then, the Uniform 
distribution has a profit value as high as 80%.
