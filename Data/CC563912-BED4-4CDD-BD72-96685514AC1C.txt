一、 前言與研究目的 
網路上其實存在著五花八門的資料，如同一個龐大的資料庫，使用者可以用不同的硬
體資源、方式來取得。但很可惜的是，縱使有如此龐大的資訊，它們都屬於「歷史資料」
的一種，也就是說，你可能找不到「現在」在某地區所發生的任何事物，或是現場沒有認
識的人可以讓你詢問。例如，你可以找到某地區的旅遊記錄，或是某場音樂會的相關訊息，
但你無法得知「目前該地區目前人潮多不多？有沒有塞車？天氣好不好？」，或是「音樂會
的內容精彩嗎？現在去還來得及嗎？」等諸如此類的問題解答。 
  因此，我們提出了 Mobile Mutual Aiding Network 的概念，希望讓目前無法被解決的問
題與請求，可以達到完善的結果。在這篇論文中，我們結合了目前熱門的技術，以及智慧
型手機等手持裝置的特色，提出 Mobile Mutual Aiding Network，簡稱 M-MAN，顧名思義，
這是一套開發於手持裝置上的互助平台，目的在於解決上述的「即時性問題」。目前我們著
重在手機上的開發，手機的特色如同前文所提，在於其分布密度遠比個人電腦的分布密度
高出許多，換句話說，幾乎每個地區每一個人都擁有手持裝置，如同無線感測器網路中的
感測器，密密麻麻的分布在某些區域裡。因此，我們可以將手機當成感測器來使用，讓它
盡可能的收集資料，再藉由網際網路提供給需要的人。M-MAN 的主要概念是：以人為中
心去發展這套系統，並且在人為因子的幫助下，取得「即時的」資訊。因此，我們提倡的
是行動互助的精神，意思是使用者可以單純的利用手機發送問題或請求，或者回覆當下的
狀況給需要的人，以達到互助的效果，就算是不認識的人也可以透過 M-MAN 來互相幫助。 
舉個例子來說，見圖表 1，某天學校舉辦了草地音樂會，某人想要知道這個音樂會好
不好聽、值不值得去參加時，他並不想親自去音樂會確認。此時他可以利用手機，或是其
他行動裝置，甚至是電腦，發送『音樂會好聽嗎?』這個問題出去，透過網路送到在音樂會
的會場裡面、或是在附近的人們，再由那些願意對這個問題做回覆的人來回答。圖中的紅
色虛線圈圈內的人們，就是在音樂會會場或是離會場很近的人們。

z 登入系統 
這個部分在每一台行動裝置重新登入系統時，都一定要啟動這個部分。因為這個部分是
系統要記錄或更新目前這個行動裝置的硬體資訊和目前使用這台機器的使用者資料到資料
庫中。因為當這台行動裝置重新登入系統時，機器的硬體資訊可能和上一次登入時的情形
不一樣。如：機器目前所在的位置不同。此外，同一台機器也有可能是不同的使用者來使
用，所以系統必須檢查並更新這台行動裝置目前是哪位使用者正在使用。 
z 詢問系統 
我們的系統的主要概念是行動互助，顧名思義就是希望大家利用手邊的行動裝置來幫
助別人。為了促使這個概念更容易達到，我們開發了『評分機制』。這個子系統是為了給那
些有即時做回覆的使用者一個獎勵的動作。 
 評分機制分為兩個，一個是扣分的機制 ，一個是加分的機制。在啟動詢問系統這個子
系統之前，會先觸發扣分的機制，因此在這我們先介紹扣分的機制，見圖 3-2。當伺服器收
到某一個使用者發送的問題時，系統會先檢查一下這位使用者的積分。如果他的積分不是
大於 0 的，那麼系統會發送通知的訊息給使用者，告知他已經無積分了。如果使用者的積
分是大於 0 的，那麼系統會對這位使用者的積分做扣分的動作後，再繼續接下來的工作。 
z 答案系統 
 這個子系統主要是在處理那些使用者回覆問題的答案。當伺服器收到使用者回覆時，
系統會先檢查這個問題是否已經截止使用者回答。如果已經截止回答了，那麼系統會回傳
訊息告知使用者。如果還沒有截止回答，系統會先把這個答案儲存在資料庫中。等到問題
截止回答後，系統會開始做分析的動作，這個分析的動作是依『分析機制』來處理答案。
這個機制主要是處理使用者回覆的答案。我們是假設當初客戶端的設計以問卷的形式。換
句話說，針對一個問題，事先製定好一些選項。因此，當使用者傳回答案時，答案的種類
就只會有當初製定的那幾種而已。那分析機制主要做的是，對每個答案做統計的動作。所
以，當伺服器回傳到使用者手上的時候，使用者即可知道，那個答案是最多人回覆的。  
z 資料搜尋系統 
 這個子系統主要是用來搜尋儲存在資料庫中的資料，包含個人資訊、歷史資料。目前
個人資訊方面，我們已經開放查詢個人的積分、個人目前所在的經緯度位置。而歷史資料
方面，則是開放查詢某一地區已經有人發問過的問題。除了這些之外，為了使用者可以更
快速了解其他的系統資訊、個人資訊…等。我們還在積極開發當中。  
z 儲存系統 
 這個子系統我們是利用了 Apache Hadoop 專案中的一個子專案，稱做『HBase』 ，它
是一個分散式的儲存系統。『HBase』它並不是關聯式的資料庫系統、它也不支援 SQL 語法，
它只有提供幾個函式來做存取資料庫裡的資料，如：Get()、Get()、Scan()。因為『HBase』
沒有支援 SQL 語法，我們無法很輕易地設計儲存資料的資料結構。 
 我們總共設計了三個不同的資料表。第一個是『machine』資料表，主要是針對某一台
裝置上的資料。如：目前這個行動裝置的經緯度位值、IP address、目前使用這台裝置來連
上系統的使用者帳號…等。第二個資料表是『personal』資料表，這個資料表主要是在記錄
每一個使用者的個人資訊，包含個人帳號、密碼、分數…等。第三個資料表是『question』，

¾ 其他功能 
(1) 提供 Mail box 的功能：將某期間內使用者發送過的問題，或是得到的回答儲存
起來，提供離線查閱的功能。 
(2) 提供更多的模組：我們希望讓更多不同的使用者需求能被滿足，因為這是一個
平台，所以能提的功能當然越多越好。因此，我們必須開發更多的模組，或是提供更
多的搜尋功能，例如多重地區搜尋。 
(3) 存取更多的硬體，例如手機上各式各樣的感測器。 
(4) 提供更多的 API。 
(5) 希望能在 Android 以外的平台開發，讓更多人可以使用這個平台。 
 
¾ 伺服端開發 
 為了找到適合的人，為了讓使用者更方便、為了能夠讓使用者自己開發，伺服端
還有很多的功能需要開發。 
 第一件是系統記錄更多的手機硬體資訊。這在選擇人選方面，更可以選到適
合的人。因為目前選擇人選只有利用地理資訊，如果系統可以把更多的手機的硬體資
訊也加入考量。例如：手機上的照相功能、目前手機的電量…等。系統為了增加回覆
率，可以避免丟訊息給只剩少許電量的手機。此外，系統有了更多的手機硬體資訊，
系統提供的功能也可以更加的多元，不再僅限於『查詢』。例如：使用者可以透過系統
找到在海邊的人，請他們分享今天的夕陽。 
 第二件是開放更多的 web API。因為我們希望我們是開發一個開放式的平台，使
用者可以利用這些 web API 開發自己專屬的網頁版行動互助系統。所以，我們正在積
極地開放更多的 web API。

¾ 技術說明 
使用者可以利用生活中最常見的手持裝置，隨時隨地取得你目前想要得知的「即
時資訊」，或是將你所知的「即時訊息」提供給需要的人，利用人與人之間的互動，達
到互相幫助的目的。因此，A-MAN 是個以人為中心的互助平台。

¾ 技術特點 
A‐MAN 是個平台而非單純的應用程式，利用模組化的概念，使用者只需要安裝一
次 A‐MAN 主程式，爾後可以依照自己的需求另外加裝其他好用的模組。所以，A‐MAN
提供的是 Relation‐based service，能夠依據使用者不同的搜尋條件，找到不同的適合
人選，這也是優於其他平台的最大特色。
